# 远程开关（4路继电器）接入计划

## 一、整体目标

将 4 路远程继电器接入现有 IoT 平台，保持框架一致性，复用报警、数据查询、可视化等能力。协议在 MQTT 层转换，平台内部统一使用 addr/addrv 格式。

---

## 二、协议定义

### 2.1 重要说明：设备不主动上报

**该设备不会主动推送数据**，平台需要通过 `getDevStatus` 主动查询获取实时状态。

---

### 2.2 继电器原生协议

#### 2.2.1 查询设备状态（平台主动拉取）

**请求（平台 → 设备）**

| 参数   | 必填 | 类型   | 说明                          |
|--------|------|--------|-------------------------------|
| method | 是   | string | 固定 `"getDevStatus"`         |
| q      | 否   | string | 过滤返回字段，如 `"slots,EMdata"` 表示不返回这些字段 |
| frameId| 否   | string | 帧 ID                         |

示例：
```json
{
  "method": "getDevStatus",
  "frameId": "1745396239780"
}
```

**应答（设备 → 平台）**

| 参数       | 类型   | 说明                                           |
|------------|--------|------------------------------------------------|
| method     | string | `"getDevStatus"`                               |
| result     | string | `"ok"` 成功，其它为失败原因                     |
| slots      | array  | 插槽状态 [slot1..slot4]，0=关/1=开             |
| imei       | string | 设备 IMEI                                      |
| netType    | string | 联网类型：4G / WiFi                            |
| signal     | int    | 信号强度 1-31                                  |
| temperature| string | 温度                                           |
| gps        | string | 经纬度，格式 "经度,纬度"                        |
| iccid      | string | 物联卡 ICCID（4G 款）                          |
| EMdata     | array  | 插槽电量计 [c, v, p] 按插槽顺序                |
| tasks      | array  | 插槽任务对象 [current, voltage, slotNum, power, status] |
| frameId    | string | 帧 ID                                          |
| timestamp  | int    | 秒级时间戳（WiFi 款不支持）                     |

示例（实际回参）：
```json
{
  "imei": "869080070209335",
  "gps": "118.0478820,024.6100616",
  "method": "getDevStatus",
  "timestamp": 1770862229,
  "signal": 29,
  "result": "ok",
  "model": "Air780E",
  "EMdata": [{"c":0,"v":0,"p":0},{"c":0,"v":0,"p":0},{"c":0,"v":0,"p":0},{"c":0,"v":0,"p":0}],
  "slots": [0, 0, 0, 0],
  "netType": "4G",
  "iccid": "898608421024D0593829",
  "tasks": [{"current":"0.000","voltage":"0.000","slotNum":1,"power":"0.000","status":"idle"}, ...],
  "temperature": "23.8",
  "frameId": "1745396239780"
}
```

#### 2.2.2 控制开关（平台 → 设备）

**请求（平台 → 设备）**

| 参数            | 必填 | 类型   | 说明                    |
|-----------------|------|--------|-------------------------|
| method          | 是   | string | 固定 `"actions"`        |
| slotNums        | 是   | array  | 插槽编号 [1,2,3,4]     |
| action          | 是   | string | `on` / `off` / `toggle` |
| hasStopDelayTask| 否   | bool   | 是否停止延时任务        |
| frameId         | 否   | string | 帧 ID                   |

示例：
```json
{
  "method": "actions",
  "slotNums": [2, 3],
  "action": "on",
  "hasStopDelayTask": false,
  "frameId": "1745396239780"
}
```

**应答（设备 → 平台，控制命令的回复）**

| 参数     | 类型   | 说明                                           |
|----------|--------|------------------------------------------------|
| method   | string | `"actions"`                                    |
| result   | string | `"ok"` 成功                                    |
| slots    | array  | 当前插槽状态 [0,1,0,1]                         |
| imei     | string | 设备 IMEI                                      |
| frameId  | string | 帧 ID                                          |
| timestamp| int    | 秒级时间戳                                     |

#### 2.2.3 平台内部格式（与 MQTT1.0/MQTT2.0 一致）

- **属性**：addr = 属性标识符，addrv = 属性值
- **远程开关映射**：slots[i] ↔ addr=switch(i+1), addrv=0|1

## 四、后端改动

### 4.1 产品表 protocol 扩展

- 新增协议值：`RELAY`（与 `MQTT1.0`、`MQTT2.0` 并列）

### 4.2 实时订阅与定时拉取

**实时订阅（已有）**  
- 平台 MQTT 客户端已订阅 `ssc/+/report`（或等价 topic）
- 下发 actions 后，设备立即将应答推到 report topic，平台实时收到
- 无需轮询响应，收到即解析入库 → 前端可快速刷新

**定时拉取（RelayStatusPoller）**  
- 设备不主动上报，需平台定时发 `getDevStatus` 拉取
- 频率可放宽：每 3 分钟遍历 protocol=RELAY 的在线设备
- 向 `ssc/{deviceCode}/command` 发布：`{ "method": "getDevStatus", "frameId": "xxx" }`
- 设备应答发布到 `ssc/{deviceCode}/report`，由 MqttMessageHandler 解析

### 4.3 MqttMessageHandler（应答解析，两条路径统一处理）

- 新增 `protocol === "RELAY"` 分支
- 应答都来自 `ssc/{deviceCode}/report`，根据 `method` 区分：
  - **路径1** `method === "actions"`：控制命令的应答，包含 slots（新状态）
  - **路径2** `method === "getDevStatus"`：定时查询的应答，包含 slots、temperature、EMdata、tasks 等
- 共通逻辑：从两种应答中提取 `slots` 数组，转换 `slots[i]` → `{ name: "switch"+(i+1), value: "0"|"1" }`
- 输出 `UnifiedReportData`，走现有 `handleUnifiedReport`
- 可选：getDevStatus 的 temperature、EMdata、tasks 等扩展为额外属性（后续迭代）

### 4.4 MqttPublisher（下发组装）

- 新增 `protocol === "RELAY"` 分支
- **控制命令**：输入 addr(switch1~4)、addrv(0|1) → 组装 `{ method, slotNums, action, frameId }`
- **查询状态**：由 RelayStatusPoller 直接组装 `{ method: "getDevStatus", frameId }` 并发布

### 4.5 初始化脚本

- 系统启动或首次部署时，检测是否存在「远程开关」产品
- 若不存在，则创建产品 + 4 属性 + 8 命令
- 可放在 `init.sql` 或独立的 `data.sql` / 启动初始化逻辑

## 五、前端改动

### 5.1 产品创建/列表

- 协议选择下拉框增加：`RELAY`（远程开关）
- 或：将「远程开关」作为预置产品，不开放自由创建，仅支持从模板创建设备

### 5.2 设备详情 - 命令控制 Tab

**产品 protocol = RELAY 时**：使用开关 UI

- 4 个 `el-switch` 或自定义开关组件
- 每个开关：label=attrName（可配置），value=实时数据中对应 addr 的值
- 点击：调用 `sendCommand({ addr, addrv })`，addrv 取当前值的反（0↔1）
- 点击后：开关进入 loading，等待设备响应
- **超时 3 秒**：若 3 秒内未收到响应、状态未更新，则提示「设备无响应」，取消 loading，保持原状态

**其它产品**：保持现有命令按钮列表

**进入命令 Tab 时**：可调用「立即拉取状态」接口，触发一次 getDevStatus，加快刷新

### 5.3 其它页面

- 实时数据、历史数据、报警配置、设备日志：**无需改动**，沿用现有逻辑
- 产品详情：属性/命令列表对远程开关只读（或仅允许修改 attrName）

## 七、实施顺序建议

1. **后端 - 协议与解析**
   - 产品表/逻辑支持 `RELAY`
   - 新增 RelayStatusPoller：每 3 分钟向 RELAY 设备发送 getDevStatus
   - MqttMessageHandler 增加 RELAY 应答解析（getDevStatus、actions 两种 method）
   - MqttPublisher 增加 RELAY 下发组装（actions 控制命令）

2. **初始化数据**
   - 编写初始化脚本：创建远程开关产品 + 4 属性 + 8 命令
   - 首次部署或升级时执行

3. **前端 - 命令控制**
   - 设备详情命令 Tab：根据产品 protocol 判断，RELAY 时渲染 4 个开关
   - 开关组件：显示状态、点击下发、loading 反馈

4. **联调与验证**
   - 模拟继电器上报/下发
   - 验证报警、历史数据、可视化是否正常

5. **产品创建流程**（可选）
   - 若采用预置产品：隐藏「创建远程开关」或改为「从远程开关模板创建设备」
   - 若允许创建：协议选 RELAY 时自动套用固定物模型

## 八、设备与继电器绑定

- 设备 `deviceCode` 需与继电器 IMEI 或厂商约定的唯一标识一致（如 `869080070209335`）
- 继电器连接平台 MQTT：
  - 订阅 `ssc/{deviceCode}/command`（接收 getDevStatus 查询、actions 控制）
  - 发布 `ssc/{deviceCode}/report`（应答 getDevStatus、actions）
- 若继电器使用独立 Broker，需增加桥接服务做 topic 与协议转换

---

## 九、设备状态获取：两条路径

设备状态（slots）有 **2 个来源**，都需解析并入库：

| 路径 | 触发方式 | 设备应答 method | 说明 |
|------|----------|-----------------|------|
| **路径1：控制响应** | 用户下发 actions 命令 | `"actions"` | 设备立即应答到 report topic，平台已订阅，实时收到 → 即时反馈 |
| **路径2：定时拉取** | 平台每 3 分钟发 getDevStatus | `"getDevStatus"` | 无人操作时保持状态更新，频率可放宽 |

两条路径的应答都发布到 `ssc/{deviceCode}/report`，MqttMessageHandler 根据 `method` 分别解析，最终都提取 `slots` 转成 switch1~4 写入存储。

---

## 十、数据流向小结

```
路径1 - 控制响应
  用户点击开关 → sendCommand(actions) → 发布到 command topic
      → 设备执行并应答到 report topic: { method: "actions", slots, result }
      → MqttMessageHandler 解析 → 存储 → 前端刷新

路径2 - 定时拉取（每 3 分钟）
  定时任务 → 发布 getDevStatus 到 command topic
      → 设备应答到 report topic: { method: "getDevStatus", slots, ... }
      → 平台已订阅 report，实时收到 → MqttMessageHandler 解析 → 存储 → 报警/展示

两条路径汇总到同一套 slots → switch1~4 存储逻辑
```
